# Dream：6 个技术亮点

这些技巧单独都不难，组合在一起说明作者在生产环境大量踩过坑。

---

## 1. 用文件 mtime 当时间戳

**普通做法：** 建一个 `last_run.json`，里面写时间。

**Dream 的做法：**
```
lock 文件的 mtime（系统自动维护的修改时间）= 上次整理时间
lock 文件的内容                           = 正在跑的进程 PID
```

一个文件存了两件事。读取只需要一次 `stat` 系统调用，极快，不用打开文件读内容。

---

## 2. 触发门按成本从低到高排序

```
第1步：读 mtime（一次 stat）      → 几乎零成本
第2步：列出会话文件列表            → 稍贵
第3步：抢锁（写文件）             → 最贵

任何一步不满足 → 立即跳出
```

每次对话结束都会跑这个检查。顺序反了成本会高很多。这叫短路求值，看起来是常识，但大多数人不会主动这样设计。

---

## 3. 崩溃恢复：保存 priorMtime 用于回滚

```
正常流程：
  抢到锁（先保存 priorMtime）→ 跑 Dream → 完成

Dream 崩溃/被杀：
  用 priorMtime 把锁的 mtime 手动改回去
  → 下次检查看到的还是"上次成功整理"的时间
  → 不会因为一次失败就永远不再触发
```

关键在于**抢锁之前就保存了 priorMtime**，失败时可以精确回滚。这是"撤销事务"的思想，用两个变量就实现了。

---

## 4. PID 复用保护

进程 ID 在操作系统里会循环复用——老进程死了，新进程可能拿到同一个 PID。

**隐患：** 锁文件里记录的 PID 还"活着"，但其实是另一个进程，Dream 就永远不会再触发了。

**解法：**
```javascript
const HOLDER_STALE_MS = 60 * 60 * 1000  // 1 小时

// 即使 PID 还活着，超过 1 小时强制认为锁过期
```

这个 bug 极其隐蔽——在生产环境跑很久才可能触发，而且没有任何报错，就是"Dream 再也不跑了"。能提前想到，说明踩过类似的坑。

---

## 5. 在 Prompt 里写"grep，不要全量读"

历史对话存成 JSONL，可能几 MB 甚至几十 MB。全量读 token 会爆炸。

**解法不是在代码里限制文件大小，而是在 Prompt 里告诉 AI：**

```
Session transcripts: (large JSONL files — grep narrowly, don't read whole files)
```

用语言约束 AI 行为，而不是代码强制限制。更灵活——AI 可以根据具体情况判断要 grep 什么关键词，而不是被硬性截断。

---

## 6. 相对时间 → 绝对时间

Prompt 里有这一句：

```
Converting relative dates ("yesterday", "last week") to absolute dates
so they remain interpretable after time passes
```

你今天在记忆里写"昨天修了登录 bug"，三个月后 Dream 读到，"昨天"是哪天？没人知道了。

让 AI 在整理时主动把"昨天"转成"2026-03-30"，记忆才有持久价值。这是大多数人根本不会想到的细节。

---

## 总结

| 亮点 | 本质 |
|------|------|
| mtime 复用 | 利用文件系统隐含信息，零额外开销 |
| 门顺序 | 短路求值，成本最小化 |
| priorMtime 回滚 | 轻量级事务，崩溃安全 |
| PID 复用保护 | 防御性编程，针对操作系统底层行为 |
| Prompt 约束读取 | 用语言控制 AI，比代码限制更柔性 |
| 绝对时间转换 | 防止记忆语义随时间腐烂 |
